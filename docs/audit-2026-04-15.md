# Siphon — Full Repository Audit (2026-04-15)

Three parallel agents reviewed the full codebase: security vulnerabilities, dead/orphaned code, and legacy patterns.

---

## CRITICAL — Fix before any public exposure

### [C-1] `.env` committed with live secrets
The actual `.env` file (not just `.env.example`) is present in the repository and contains:
- Fernet encryption key: `aMgwZ5VXJPcxLd3zoM510Dtb59-O6HhCUG8DhlZUiAo=`
- JWT secret: `dev-jwt-secret-change-in-production`
- Admin credentials: `admin@example.com` / `changeme123`
- Database URL with password

Anyone with read access can decrypt all stored connection credentials immediately.

**Fix:** Add `.env` to `.gitignore`, rotate the Fernet key, JWT secret, and all credentials.

---

### [C-2] `.env.example` ships a real, valid Fernet key
`SIPHON_ENCRYPTION_KEY=aMgwZ5VXJPcxLd3zoM510Dtb59-O6HhCUG8DhlZUiAo=` is not a placeholder — it is a functional key. Any deployment that copies `.env.example → .env` without generating a fresh key uses a publicly known key.

**Fix:** Replace with `SIPHON_ENCRYPTION_KEY=REPLACE_ME_run_python_-c_"from_cryptography.fernet_import_Fernet;print(Fernet.generate_key().decode())"`.

---

### [C-3] Weak JWT default is silently functional in dev mode
`src/siphon/auth/jwt_utils.py:12–20` — the default signing secret `"dev-secret-change-in-production"` is live whenever `SIPHON_JWT_SECRET` is not set. The `.env` ships this default, so any attacker who reads the source or `.env` can forge valid JWT tokens for any user and role.

**Fix:** Reject startup if `SIPHON_JWT_SECRET` equals the default, regardless of `SIPHON_DEV_MODE`.

## HIGH — Security

### [H-S1] SSRF: `HTTPRestSource` has no host validation
`src/siphon/plugins/sources/http_rest.py` — `extract_batches()` and `_fetch_oauth2_token()` make outbound HTTP requests to caller-supplied `url` and `token_url` without calling `_validate_host`. An admin can point a pipeline at `http://169.254.169.254/latest/meta-data/` or any internal service.

**Fix:** Call `_validate_host(self.url)` at the start of `extract_batches()` and `_fetch_oauth2_token()`.

---

### [H-S2] SSRF: `webhook_url` on pipelines is not validated
`src/siphon/worker.py:137–147` — on every job completion, `_fire_webhook` POSTs to the pipeline's `webhook_url` without host validation. An admin can probe internal services and exfiltrate job metadata.

**Fix:** Validate `webhook_url` against `SIPHON_ALLOWED_HOSTS` at pipeline creation or before firing.

---

### [H-S3] SSRF guard is opt-in (fail-open by default)
`src/siphon/plugins/sources/sql.py:147–148` — `_validate_host` returns immediately when `SIPHON_ALLOWED_HOSTS` is empty. The default config (including `.env` as shipped) has zero SSRF protection for SQL connections.

**Fix:** Fail-closed: either ship a safe default allowlist, or refuse startup without `SIPHON_ALLOWED_HOSTS`.

---

### [H-S4] Raw DB exception messages returned to callers
- `src/siphon/preview/router.py:75` — `raise HTTPException(400, f"Query failed: {exc}")` exposes database internals (table names, column names, server version) to any authenticated operator via the preview endpoint.
- `src/siphon/connections/router.py:135,248` — both test endpoints return raw `connectorx`/`paramiko` exception text.

**Fix:** Log `exc` server-side, return `"Query failed — check server logs"` to the caller.

---

### [H-S5] `POST /connections/test` (unsaved) accessible to all authenticated principals
`src/siphon/connections/router.py:122` — accepts raw `type + config` from any authenticated user (not just admins) and makes outbound network connections. Operators who cannot create connections can still probe arbitrary hosts through this endpoint.

**Fix:** Add `principal.require_admin()` to `test_connection_unsaved`.

---

### [H-S6] PostgreSQL password hardcoded in `docker-compose.yml`
`docker-compose.yml:66–68` — `POSTGRES_PASSWORD: siphon` is a literal, not `${POSTGRES_PASSWORD}`. It cannot be rotated through `.env`.

**Fix:** Change to `POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}` and add the var to `.env.example`.

---

## HIGH — Broken features (runtime bugs)

### [H-B1] `GET /runs/{id}/logs` always returns empty — logs are permanently inaccessible
`src/siphon/runs/router.py:89–100` — `queue.get_job()` always returns `None` in the Celery-backed queue (by design, per the comment in `queue.py:41`). The `LogViewer` component polls this endpoint every 2s during active runs and always receives `[]`. Job logs written by `worker.py` into `job.logs` live in Celery worker process memory and are never surfaced.

**Fix:** Persist per-job log lines to `job_runs.logs JSONB` during execution, then read from DB in the logs endpoint.

---

### [H-B2] `POST /extract` lies about being synchronous
`src/siphon/main.py:288–310` — the route is named "synchronous extraction — blocks until done" but after Phase 16 it dispatches to Celery and returns immediately with `status: "queued"`. Fields `rows_read`, `rows_written`, `duration_ms`, and `logs` are always `null`/`[]`.

**Fix:** Either remove the endpoint (it's gated behind `SIPHON_ENABLE_SYNC_EXTRACT=true`) or update the docstring and response contract to reflect async behavior.

---

### [H-B3] `GET /jobs/{job_id}/logs` permanently returns 404
`src/siphon/main.py:344–352` — comment says "Task 4 will implement DB-backed lookup; for now return 404." Phase 17 is complete; this was never finished. The endpoint is in the OpenAPI schema and referenced in `docs/api-reference.md`.

**Fix:** Implement DB-backed log retrieval or remove the endpoint and its `LogsResponse` import.

---

### [H-B4] `_update_gdpr_event` reuses global session factory inside Celery worker
`src/siphon/tasks.py:221–236` — `_update_gdpr_event` calls `get_session_factory()` which returns the module-level session factory whose engine was bound to the parent process event loop. This is the exact "Future attached to a different loop" bug that `_run_job_async` already documents and works around by creating a fresh engine per task. `_update_gdpr_event` doesn't follow the same pattern.

**Fix:** Create a fresh engine in `_update_gdpr_event` exactly as `_run_job_async` does, and call via `asyncio.run()`.

---

### [H-B5] `asyncio.get_event_loop()` called inside running async context — deprecated, will error in Python 3.12+
- `src/siphon/connections/router.py:130,244`
- `src/siphon/scheduler.py:323`

Calling `asyncio.get_event_loop()` inside an `async def` raises `DeprecationWarning` since Python 3.10 and will become an error in a future Python version.

**Fix:** Replace with `asyncio.get_running_loop()` in the router handlers. For `scheduler.py`, store the loop at startup and pass it explicitly.

---

## MEDIUM — Security

### [M-S1] Watermark interpolated into SQL with ad-hoc escaping
`src/siphon/pipelines/watermark.py:82–90` — `last_watermark` is escaped with `.replace("'", "''")` and interpolated as a SQL string. Server-generated today, but fragile: dialect-dependent and incomplete. If a future API allows resetting the watermark, this is an injection vector.

**Fix:** Use parameterized binding or validate `watermark` with a strict ISO-8601 regex before interpolation.

---

### [M-S2] S3 purge date filter uses string lexicographic comparison
`src/siphon/tasks.py:180` — `f < base_path + f"/_date={date_str}"` compares S3 object paths as strings. Not a reliable date comparison.

**Fix:** Extract the date component from each path with a regex and compare as `date` objects.

---

### [M-S3] Internal details leaked from `queue.py` and `pipelines/router.py`
`src/siphon/queue.py:65` and `src/siphon/pipelines/router.py:481` — `str(exc)` included directly in HTTP error responses. May expose Redis URL or Python internals.

**Fix:** Log full exception, return stable user-facing message.

---

## MEDIUM — Technical debt

### [M-T1] `structlog` called with `%s` printf style throughout (18+ occurrences)
The project uses `logger.info("event", key=value)` correctly in `worker.py` but uses `logger.info("Found %d files", count)` in:
- `src/siphon/scheduler.py` (18+ occurrences)
- `src/siphon/plugins/sources/sftp.py` (8+ occurrences)
- `src/siphon/plugins/sources/sql.py`
- `src/siphon/plugins/destinations/s3_parquet.py`
- `src/siphon/plugins/sources/http_rest.py`
- `src/siphon/utils/retry.py`

This defeats structured logging — JSON output contains a pre-formatted string instead of queryable fields.

**Fix:** `logger.info("files_found", count=n)` instead of `logger.info("Found %d files", n)`.

---

### [M-T2] `Connection.max_concurrent_jobs` stored and surfaced but silently ignored at runtime
`src/siphon/orm.py:49` — the column is persisted and returned by the API, but `queue.py:52` accepts it "for API compatibility" and does nothing with it. Operators who configure it expect an effect.

**Fix:** Either implement the concurrency check against `job_runs WHERE status='running'` in the worker, or remove the column and API field.

---

### [M-T3] Soft-deleted resources block name reuse
`src/siphon/pipelines/router.py:215` and `src/siphon/connections/router.py:97` — uniqueness checks do not filter `deleted_at IS NULL`. A soft-deleted pipeline/connection permanently occupies its name slot. The DB-level `UNIQUE` constraint will cause a 500 IntegrityError on reuse.

**Fix:** Add `Model.deleted_at.is_(None)` to the uniqueness query in both `create_pipeline` and `create_connection`.

---

### [M-T4] `GET /health/ready` always returns OK — Celery broker not checked
`src/siphon/main.py:362` — `queue.is_draining` and `queue.is_full` are hardcoded `False` in the Celery queue. The health endpoint gives no signal about whether the broker (Redis) is reachable or workers are available.

**Fix:** Ping Redis in the readiness check (e.g., `celery_app.control.ping(timeout=1)`).

---

### [M-T5] `purge_s3_data_task` uses `asyncio.new_event_loop()` instead of `asyncio.run()`
`src/siphon/tasks.py:214–218` — inconsistent with `run_pipeline_task` which uses `asyncio.run()`. The old pattern doesn't call `loop.shutdown_asyncgens()` before close, risking resource leak warnings in Celery workers.

**Fix:** Replace with `asyncio.run(_update_gdpr_event(event_id, result))` (after fixing H-B4).

---

## LOW — Cleanup

### [L-1] Dead parameter: `purge_s3_data_task(db_url)` — passed as `None`, never used
`src/siphon/tasks.py:195` — parameter accepted over the wire via Celery, silently ignored.

`src/siphon/orm.py:1–2` — `import uuid` and `import uuid as uuid_module` are both present. The alias is only used in `GdprEvent` but `uuid.uuid4` would work identically.

### [L-4] `SQLSource.extract()` unreachable in production
`src/siphon/plugins/sources/sql.py:65–75` — `SQLSource.extract_batches()` overrides the ABC's default, so the base `extract()` fallback is never triggered.

### [L-5] `_make_job_and_plugins` constructs source/dest objects then discards them
`src/siphon/main.py:268–269` — both plugin objects are assigned and immediately thrown away (`job, _, __ = _make_job_and_plugins(req)`) on every call site.

### [L-6] `ExampleParser` stub registered in production autodiscovery

### [L-7] `rows_failed` in TypeScript `JobRun` has no backend counterpart
`frontend/src/lib/api.ts:150` — `rows_failed: number | null` is typed but the backend never sends this field. Will always be `undefined` at runtime.

---

### [L-8] `queue.start()` / `queue.drain()` called in lifespan — both are no-ops
`src/siphon/main.py:176,182` — orphaned from the asyncio-queue era. Safe to remove.

## DOCS DRIFT

### `docs/architecture.md` describes Phase 2, not Phase 16
The following sections are entirely wrong after the Celery migration:
- **Line 22** — diagram labels queue as `(asyncio + ThreadPoolExecutor)`, worker as `(blocking I/O thread)`
- **Lines 60–86** — request lifecycle describes asyncio queue path
- **Lines 132–140** — "In-memory queue + thread pool (not Celery/Redis)" section explicitly states "There is no Redis, no Celery, no broker" — the opposite of the current architecture
- **Line 108** — directory listing describes `queue.py` as "In-memory job queue + eviction loop"

Any new contributor reading this doc will have a completely wrong mental model of the system.

**Fix:** Update diagram, request lifecycle, and design decisions section to reflect Celery + Redis architecture.
