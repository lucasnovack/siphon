# Siphon ‚Äî Phases 15‚Äì17 Design

**Date:** 2026-04-11  
**Status:** Approved  
**Scope:** Scale, Performance, Compliance ‚Äî Parquet-only destination

---

## Context

Phases 1‚Äì14 delivered the full extraction platform: sources (SQL, SFTP, HTTP/REST), S3/Parquet destination, parsers (CSV, JSON, Avro), scheduling, watermarks, data quality, schema registry, PII masking, backfill, webhooks, structured logging, OTEL tracing, and data lineage.

Phases 15‚Äì17 focus exclusively on **Scale, Performance, and Compliance**. Out of scope: deployment (K8s, Airflow operator), streaming (Kafka), alternative formats (Delta Lake, Iceberg), additional destinations.

### Simplification
BigQuery and Snowflake destinations are removed. S3/Parquet is the only destination going forward. This simplifies the codebase, removes maintenance burden, and reflects the tool's identity as a Bronze-layer EL tool.

## Phase 15 ‚Äî Cleanup + Performance

### 1. Remove BigQuery and Snowflake Destinations

Delete:
- `src/siphon/plugins/destinations/bigquery_dest.py`
- `src/siphon/plugins/destinations/snowflake_dest.py`

Update:
- `models.py` ‚Äî remove `BigQueryDestinationConfig`, `SnowflakeDestinationConfig`
- `plugins/destinations/__init__.py` ‚Äî remove from registry
- `GET /connections/types` response ‚Äî remove both types
- All associated tests

Only `S3ParquetDestination` remains.

### 2. Per-Source Concurrency Limiting (Connection Pooling)

ConnectorX is a bulk-extraction library ‚Äî individual connection pooling adds little value for long-running queries. The real risk is fan-out: multiple pipelines pointing at the same source firing concurrently and overwhelming it.

**Solution:** add `max_concurrent_jobs: int` (default `2`) to the `Connection` model and ORM. The worker checks, before starting a job, how many active jobs share the same `source_connection_id`. If at or above the limit, the job is re-queued with exponential backoff (max 3 retries before marking `failed`).

Implementation uses the existing `_jobs` dict (Phase 15) and migrates to Redis/Celery state in Phase 16.

Additionally: `HTTPRestSource` creates a new `httpx.AsyncClient` per request today. Replace with a module-level singleton `AsyncClient` (configured with timeouts and retries), closed on app shutdown.

**Schema changes:**
- `connections`: add `max_concurrent_jobs INTEGER NOT NULL DEFAULT 2`
- Migration: `008_add_connection_concurrency.py`

**API changes:**
- `ConnectionCreate` / `ConnectionUpdate` / `ConnectionResponse`: add `max_concurrent_jobs`
- Frontend: add field in `ConnectionForm`

### 3. Job Prioritization

**Model change:** add `priority` enum (`low`, `normal`, `high`, default `normal`) to `pipelines` table.

**Queue change:** replace `asyncio.Queue` with `asyncio.PriorityQueue` in `queue.py`. Each enqueued item is a tuple `(priority_int, enqueued_at_iso, job)` where `high=0, normal=1, low=2`. Timestamp as tiebreaker prevents starvation.

**Worker:** no changes needed ‚Äî worker pulls from queue the same way.

**API changes:**
- `PipelineCreate` / `PipelineUpdate` / `PipelineResponse`: add `priority`
- `POST /pipelines/{id}/trigger`: priority inherited from pipeline
- Frontend: priority dropdown in `PipelineWizard` (step 4) and `PipelineEditPage`

**Schema changes:**
- `pipelines`: add `priority VARCHAR(10) NOT NULL DEFAULT 'normal'`
- Migration: `009_add_pipeline_priority.py`

### Testing
- Concurrency limit: mock `_jobs` with N active jobs on same connection, assert new job is requeued
- Priority queue: enqueue high/normal/low jobs, assert dequeue order
- Cleanup: assert BigQuery/Snowflake types absent from `/connections/types`

## Phase 16 ‚Äî Celery + Redis (Horizontal Scale)

### Architecture

Replace `asyncio.Queue` + `ThreadPoolExecutor` (in-process, single pod) with Celery workers (separate processes, multi-pod capable).

```
FastAPI (API pod)
    ‚îÇ  enqueue via Celery
    ‚ñº
Redis (broker + result backend)
    ‚îÇ  consume
    ‚ñº
Celery Worker (1..N pods)
    ‚îÇ  execute
    ‚ñº
run_job() in worker.py  ‚îÄ‚îÄ‚îÄ PostgreSQL (job_runs)
```

### New files

**`src/siphon/celery_app.py`**
- `Celery("siphon", broker=REDIS_URL, backend=REDIS_URL)`
- Serializer: JSON
- Three queues: `high`, `normal`, `low` (map to pipeline priority)
- `task_acks_late=True` ‚Äî ack after completion, not before (at-least-once delivery)
- `worker_prefetch_multiplier=1` ‚Äî one task at a time per worker process

**`src/siphon/tasks.py`**
- `@celery_app.task(bind=True, max_retries=3) run_pipeline_task(self, job_dict)`
- Deserializes `Job` from dict, calls `run_job(job)`, handles exceptions with `self.retry()`
- On terminal failure: marks `job_runs` as `failed`

### Migration from in-memory state

`_jobs: dict` in `queue.py` is not shareable across processes. In Phase 16:
- Job state lives in `job_runs` table (already exists)
- `GET /jobs/{id}` reads from DB instead of memory
- `POST /jobs/{id}/cancel` sends Celery `revoke(terminate=True)` + updates DB
- Active job count for concurrency limiting reads from `job_runs WHERE status = 'running' AND source_connection_id = ?`

`queue.py` becomes a thin wrapper: `enqueue(job)` ‚Üí `run_pipeline_task.apply_async(args=[job.dict()], queue=job.priority)`

### Graceful drain
`SIGTERM` ‚Üí Celery worker finishes current task before exiting (`--without-gossip --without-mingle --without-heartbeat` for clean shutdown). `terminationGracePeriodSeconds` in docker-compose set to 300s (matches max expected job duration).

### Infrastructure
`docker-compose.yml` additions:
- `redis:7-alpine` service
- `siphon-worker` service: same image, entrypoint `celery -A siphon.celery_app worker --loglevel=info -Q high,normal,low`

New env vars: `REDIS_URL` (default `redis://localhost:6379/0`).

### What does NOT change
`worker.py` internals (`run_job`, `_sync_extract_and_write`, `_persist_job_run`, etc.) are preserved. Celery is the transport layer only.

### Testing
- Task serialization: `Job` ‚Üí dict ‚Üí `Job` roundtrip
- Priority routing: high-priority job dispatched to `high` queue
- Concurrency limit: reads from DB `job_runs` instead of in-memory dict
- Graceful shutdown: worker acks late, in-flight task completes after SIGTERM

## Phase 17 ‚Äî GDPR Compliance

### 1. Soft Delete

Add `deleted_at TIMESTAMP WITH TIME ZONE` (nullable) to: `connections`, `pipelines`, `schedules`, `users`.

Migration: `010_add_soft_delete.py`

**Behavior:**
- All `GET` list endpoints filter `WHERE deleted_at IS NULL`
- `GET /{id}` returns 404 if `deleted_at IS NOT NULL`
- `DELETE` endpoints set `deleted_at = now()` instead of `DELETE FROM`
- Cascade: soft-deleting a `Connection` soft-deletes all `Pipeline`s referencing it as source or destination, and removes their schedules from Celery

Historical data is preserved: `job_runs` and lineage records remain intact after soft delete, queryable via admin endpoints.

### 2. S3 Purge API

**Endpoint:** `DELETE /api/v1/pipelines/{id}/data`

Query params:
- `before: date` (optional) ‚Äî delete files from partitions before this date
- `partition: str` (optional) ‚Äî delete a specific partition value

Admin-only. Requires pipeline to exist (even if soft-deleted).

**Flow:**
1. Load pipeline's `destination_path` from DB
2. List matching Parquet files via `PyArrow S3FileSystem`
3. Delete files
4. Insert record into `gdpr_events` table
5. Return sync response with summary `{files_deleted, bytes_deleted}`

If file count exceeds 1000, the purge becomes a background Celery task and the API returns `202 Accepted` with a task ID for polling.

**New table: `gdpr_events`**
```
id UUID PK
pipeline_id UUID FK pipelines.id
requested_by UUID FK users.id
before_date DATE nullable
partition_filter TEXT nullable
files_deleted INTEGER
bytes_deleted BIGINT
status VARCHAR(20)  -- completed | failed | in_progress
requested_at TIMESTAMPTZ
completed_at TIMESTAMPTZ nullable
```

Migration: `011_add_gdpr_events.py`

**Endpoints:**
- `DELETE /api/v1/pipelines/{id}/data` ‚Äî initiate purge (admin-only)
- `GET /api/v1/gdpr/events` ‚Äî list purge history (admin-only, paginated)
- `GET /api/v1/gdpr/events/{id}` ‚Äî status of a specific purge

### Testing
- Soft delete: `DELETE /connections/{id}` ‚Üí 404 on `GET`, pipelines cascaded
- Purge sync: mock S3, assert files listed + deleted + event recorded
- Purge async: >1000 files ‚Üí 202 + Celery task created
- Audit: `GET /gdpr/events` returns purge records
- Non-admin: 403 on all GDPR endpoints
